Performance vs. Flexibility
nel Datapath IPA/eBPF

IPA/eBPF non ha una sola implementazione naturale. Esiste uno spazio di progetto in cui si puo scegliere quanto specializzare il modello nel datapath e quanto invece mantenere flessibilita runtime.

Domanda di ricerca: Quanto costa rendere IPA piu flessibile nel datapath?
La valutazione misura sia la datapath performance che il control-plane flexibility.

Control-plane flexibility

Costo di Aggiornamento Modello

La dimensione sperimentale piu interessante: quanto costa cambiare modello a runtime per ciascuna pipeline?

1. Hardcoded

Procedura di aggiornamento
clang -O2 -target bpf model_new.c -o model_new.o ip link set dev eth0 xdp obj model_new.o sec xdp

Richiede: generazione sorgente C, compilazione LLVM, reload XDP. Il datapath e interrotto durante il reload.

2. Template architetturale

Procedura di aggiornamento
bpf_map_update_elem(arch_weights, &key, &new_w, 0) bpf_map_update_elem(model_registry, &mid, &info, 0)

Solo map update dal userspace. Nessuna interruzione del datapath. Aggiornamento atomico per pacchetti in volo.

3. Modular pipeline

Procedura di aggiornamento
bpf_map_update_elem(layer_weights, &key, &w, 0) bpf_map_update_elem(layer_chain, &idx, &fd, 0)

Aggiornamento pesi + eventuale modifica della sequenza layer. Permette di cambiare anche l'architettura (non solo i pesi).

Formulazione da paper

Formulazione Accademica

Testo pronto per la sezione "Design Space" del report/paper (dal professore).

Paper formulation

We consider three implementation points in the IPA/eBPF design space. The first one hardcodes each neural model into a dedicated eBPF program, maximizing datapath performance at the cost of requiring code regeneration and program reloading for each model update. The second one relies on pre-built architectural templates, where each eBPF program implements a common neural architecture and retrieves model-specific quantized parameters from BPF maps. This reduces recompilation needs while preserving a statically verifiable inference structure. The third one decomposes neural inference into reusable eBPF layer modules connected through tail calls, using a per-CPU scratch map to exchange intermediate activations. This maximizes architectural flexibility, but introduces additional tail calls and map accesses, thus reducing the maximum achievable packet processing rate.

Riscontro pratico

Vincoli del Verifier eBPF

Il Livello 1 (hardcoded) e quello dove "massime prestazioni, zero lookup" e piu difficile da ottenere in pratica: il verifier impone due vincoli che collidono con l'idea di codice completamente unrolled.

▮

Stack limit: 512 B

Feature vector o tabelle di pesi per-neurone troppo grandi eccedono il budget di stack di un programma eBPF.

⋮

Esplosione del CFG

Ramificare (switch/if) su un valore runtime dentro un ciclo per-neurone moltiplica (non somma) i path che il verifier deve esplorare.

⊞

BCC e dati globali

Senza CO-RE, BCC non rilocca correttamente array static const per XDP: l'indirizzo puo collassare a 0 al load.

Case study — Pipeline 1, fc1 (65-4-4-7)

Solo 3 feature sono vive per pacchetto (ttl, iface one-hot, node one-hot): il peso associato dipende quindi da un indice runtime limitato (_iface ∈ 1..6, _node ∈ 0..51). Due tentativi sono stati scartati prima della soluzione attuale:

  1. switch(_iface)+switch(_node) per neurone — con 4 neuroni i path esplorati crescono come (7·52)4 ≈ 1.75×1010, ben oltre il budget di 1.000.000 istruzioni del verifier → Permission denied.
  2. Array static const per neurone, indicizzati con variabile mascherata — niente esplosione di path, ma BCC compila questi array come dati globali non rilocati per XDP: la load risultante legge dall'indirizzo 0 → R1 invalid mem access 'scalar'.
  3. Fix adottato: un solo switch(_iface) e un solo switch(_node) per l'intero programma, dove ogni case assegna il contributo per tutti i neuroni insieme. I branch restano O(7+52) indipendentemente dal numero di neuroni, e i valori restano scalari su stack (64 B totali) — nessuna global, nessuna map, preservando la proprieta "zero lookup pesi" del Livello 1.

Dato sperimentale interessante per il paper: il costo della specializzazione massima non e solo prestazionale, ma anche di ingegneria — richiede una codifica del branching ad-hoc per non violare i limiti del verifier, mentre i Livelli 2 e 3 evitano il problema a monte delegando i pesi a BPF_ARRAY map.
