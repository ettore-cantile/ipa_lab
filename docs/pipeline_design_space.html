Design Space Analysis

Performance vs. Flessibilità
nel Datapath IPA/eBPF

IPA/eBPF non ha una sola implementazione naturale. Esiste uno spazio di progetto in cui si può scegliere quanto specializzare il modello nel datapath e quanto invece mantenere flessibilità runtime.

Domanda di ricerca: quanto costa rendere IPA più flessibile nel datapath?
La valutazione misura sia la datapath performance che il control-plane flexibility — non solo throughput, ma anche il costo di aggiornamento del modello e la gestione dello stato delle interfacce.

Design Space IPA/eBPF

Tre livelli principali di implementazione. Aumentando la flessibilità: diminuisce la specializzazione del codice, aumentano map lookup, tail calls e passaggio di stato, diminuiscono le prestazioni massime.

Gestione dello stato
Stato delle Interfacce nel Datapath

Ogni pipeline deve risolvere un problema comune: come propagare nel datapath eBPF l'informazione sullo stato delle interfacce (link-state, mapping ifindex → classe, one-hot encoding dei vicini) senza violare i vincoli del verifier.

Stato letto dal control plane

Aggiornamento tramite BPF map

Il control plane Python mantiene la tabella degli stati delle interfacce (link_state[ifindex] → {up, bw, neighbor_id}) e la scrive in una BPF_HASH map condivisa con tutti i programmi eBPF delle tre pipeline.

Quando FRR notifica un cambio di stato (link up/down, cambio metrica), il controller aggiorna la map lato userspace con bpf_map_update_elem. I programmi eBPF leggono lo stato aggiornato al prossimo pacchetto processato, senza necessità di reload.

/* lato userspace (control plane) */ iface_map[ifindex] = { .state = LINK_UP, .neighbor = node_id, /* 0..51 */ .iface_idx = iface_cls /* 1..6 */ };

One-hot encoding nel datapath

Come le feature vengono costruite in kernel space

Il programma eBPF legge dalla map l'iface_idx e il neighbor del pacchetto corrente e li converte in one-hot encoding compatibile con il modello addestrato (FRR_model.py):

6 link-state [0..5] + 6 iface one-hot [6..11] + TTL [12] + 52 node one-hot [13..64]. Questo vettore da 65 feature viene passato all'inferenza della pipeline attiva.

/* feature vector layout (65 elementi) */ fv[iface_idx + 6] = 1; /* iface one-hot */ fv[neighbor + 13] = 1; /* node one-hot */ fv[12] = ttl; /* link-state [0..5]: da iface_map */

Pipeline 1 — switch sui valori runtime

Vincolo verifier: nessun array statico indicizzato

Nel codice hardcoded i valori iface_idx e neighbor sono variabili runtime. Il verifier eBPF non permette di indicizzare array static const con variabili runtime (la relocation fallisce e l'accesso collassa a indirizzo 0).

Soluzione adottata: un solo switch(iface_idx) e un solo switch(neighbor) per l'intero programma, dove ogni case assegna il contributo per tutti i neuroni insieme. I branch restano O(6 + 52) indipendentemente dal numero di neuroni, e i valori restano scalari su stack — nessuna global, nessuna map, zero lookup pesi.

Pipeline 2 e 3 — lookup da BPF_ARRAY

Flessibilità tramite map, senza vincoli sulle variabili

Nelle pipeline 2 e 3 i pesi sono in BPF_ARRAY. L'accesso tramite indice aritmetico (non tramite array static const) è sicuro per il verifier: la map è un oggetto kernel registrato, non un puntatore non rilocato.

L'encoding one-hot viene costruito allo stesso modo (stessa iface_map condivisa), ma il prodotto scalare con i pesi avviene tramite bpf_map_lookup_elem(&weight_map, &offset), eliminando completamente il problema di relocation e permettendo di cambiare i pesi a runtime senza toccare il codice eBPF.

Perché non serve una fwd_table dedicata nelle tre pipeline?
Le pipeline si occupano esclusivamente di classificazione: ricevono un pacchetto, estraggono le feature, eseguono l'inferenza e producono un'azione (drop / pass / redirect verso un ifindex). Il mapping classe → ifindex di uscita è risolto inline nella pipeline stessa (switch su best_cls nella P1, lookup su action_map nelle P2/P3) — non è necessaria una tabella di forwarding separata. Una fwd_table diventerebbe rilevante solo se IPA dovesse gestire routing IP generico, che non è nel perimetro di queste tre pipeline.
